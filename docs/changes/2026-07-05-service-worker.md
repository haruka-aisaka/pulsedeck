# Service Worker による静的アセットのキャッシュ

## 背景

起動のたびに HTML/CSS/JS をネットワークから取得しており、起動速度が回線に依存する。
オフライン（サーバー停止・圏外）ではアプリの殻すら開けない。

## 現状の挙動

- 毎回ネットワークからアセットを取得
- サーバー不達時はブラウザのエラーページが表示される

## 変更後の挙動

- 初回アクセス時に Service Worker が静的アセット（HTML・CSS・JS・manifest・アイコン）をプリキャッシュする
- 2 回目以降はキャッシュから即座に表示し、裏で最新版を取得してキャッシュを更新する
  （stale-while-revalidate。デプロイ後は「次の起動」で新 UI に切り替わる）
- サーバー不達時もアプリの殻（スプラッシュ → 切断表示のダッシュボード）が開く
- `/api/*` はキャッシュ対象外（常にネットワーク直行。古いメトリクスを見せない）

## 制約（仕様として明記）

- Service Worker は secure context (HTTPS または localhost) でのみ動作する。 `http://<host>:8480`
  の平文アクセスでは登録されず、従来どおりの挙動になる （`tailscale serve` の HTTPS 経由なら有効）

## スコープ外

- メトリクスデータ（スナップショット・履歴）のクライアント保存
- 更新検知時の「新しいバージョンがあります」トースト表示

## E2E テストチェックリスト

- ### 正常系
  - [ ] 初回ロードで SW が登録され、リロード後にページが SW 配下になる
  - [ ] 静的アセットが CacheStorage に入っている
- ### 異常系
  - [ ] オフライン状態でもページが開き、切断表示になる
  - [ ] `/api/*` はキャッシュされない
- ### エッジケース
  - [ ] 非 secure context（平文 HTTP）でもエラーなく従来どおり動く
